# План реализации Telegram Mini App
### Маркетплейс блогеров и бизнеса · рабочий документ для сборки

Этот документ — «карта» проекта. Держи его в корне репозитория (`PLAN.md`) и
скармливай Claude Code по частям (раздел = задача). По мере решений обновляй его
и веди отдельный `DECISIONS.md`.

---

## 1. Как мы работаем втроём

| Роль | Кто | Зона ответственности |
|---|---|---|
| **Планировщик / архитектор** | Claude (чат) | Архитектура, схема данных, разбивка на задачи, тех-задания (ТЗ) для Claude Code, критерии приёмки, ревью логики |
| **Оркестратор** | Ты | Решения в точках выбора, аккаунты/ключи/деньги, юридическая часть, тестирование в реальном Telegram, приёмка |
| **Реализатор** | Claude Code | Пишет код, тесты, запускает, чинит, деплоит по ТЗ |

**Цикл одной задачи:**
1. Claude даёт точное ТЗ на один вертикальный срез (одна фича от UI до БД) + критерии «готово».
2. Ты читаешь, утверждаешь/правишь решения, передаёшь ТЗ в Claude Code.
3. Claude Code реализует, пишет тесты, запускает, показывает как проверить.
4. Ты тестируешь **в реальном Telegram**, сообщаешь результат.
5. Claude корректирует следующее ТЗ. Повтор.

**Принципы:**
- **Вертикальные срезы.** Одна фича целиком (экран → API → БД → бот), а не «сначала весь фронт».
- **Git с первого дня.** Ветка на фичу, ревью диффов, маленькие коммиты.
- **Тест в Telegram рано.** Самообман «локально работает» здесь дорог — проверяй в мессенджере.
- **Рискованное — вперёд.** Аутентификация и платежи/эскроу делаются раньше украшений.
- **Живые документы.** `PLAN.md` (этот), `SPEC` на фичу, `DECISIONS.md` (твои решения), `.env.example` (список секретов).

## 2. Архитектура (как всё связано)

```
Telegram (клиент)
   │
   ├── Mini App (веб-фронтенд, открывается из бота)
   │        └── авторизация через initData → Backend API
   │
   └── Bot (вход в приложение, уведомления, кнопка «Открыть»)
            │
        Backend API (бизнес-логика, состояния лотов, эскроу)
            │
   ┌────────┼─────────────┬──────────────┐
PostgreSQL  Redis      Платёжный        Админка
(данные)  (очереди/    провайдер     (споры, выплаты,
          кэш, позже)  (PSP)          модерация)
```

- **Mini App** — это обычное веб-приложение в WebView Telegram. Обязателен HTTPS с настоящим сертификатом (самоподписанные не грузятся на iOS/Android).
- **Аутентификация** — фронт передаёт `initData`, бэкенд **криптографически проверяет подпись** (обязательно, иначе любой подделает пользователя).
- **Бот** — точка входа и канал уведомлений (в т.ч. «Хочу такой проект»).

## 3. Технологический стек

**Рекомендация по умолчанию — TypeScript на всём стеке.** Для сборки в одиночку с
AI это проще всего: один язык, общие типы между фронтом и бэком, Claude Code с ним
работает отлично.

| Слой | Выбор | Почему |
|---|---|---|
| Фронтенд | **React + TypeScript + Vite**, `@telegram-apps/sdk-react`, UI-кит `@telegram-apps/telegram-ui` | Официальный стек TMA; UI-кит даёт нативный вид Telegram «из коробки». Старт — из шаблона `Telegram-Mini-Apps/reactjs-template` |
| Бот | **grammY** (Node/TS) | Современный, типизированный, простой |
| Бэкенд | **NestJS** (структура) или **Fastify** (легче) | Для MVP бота и API можно держать в одном сервисе |
| ORM | **Prisma** | Типобезопасный, отличный DX, удобен Claude Code |
| БД | **PostgreSQL** | Реляционные связи (пользователи, лоты, отклики, транзакции, отзывы) |
| Кэш/очереди | **Redis** | Очередь уведомлений, rate-limit. На старте можно отложить |
| Хостинг фронта | Cloudflare Pages / Vercel | Бесплатно, HTTPS, CDN |
| Хостинг API+бота | Railway / Render / Fly.io или VPS | Нужен постоянный процесс + HTTPS |
| Managed Postgres | Neon / Supabase / Railway | Не поднимать БД руками |
| Локальная разработка | `cloudflared` или `ngrok` (HTTPS-туннель), BotFather | Чтобы тестить Mini App в Telegram с локалки |

**Альтернатива:** Python (`FastAPI` + `aiogram` + `SQLAlchemy/Alembic`) — тоже
полностью рабочий вариант. Главное — **выбрать один стек и не смешивать.**

> **Решение оркестратора (Фаза 0):** TS-стек или Python-стек.

Структура монорепо (предлагаемая):
```
/apps/miniapp   — фронтенд (Mini App)
/apps/api       — бот + backend API
/packages/types — общие типы (TS)
PLAN.md  DECISIONS.md  .env.example
```

## 4. Модель данных (черновая схема)

Сущности и ключевые поля — основа для Claude Code:

- **User** — `telegram_id`, `role` (`blogger`/`company`), `status`, `created_at`
- **BloggerProfile** — `user_id`, `display_name`, `bio`, `categories[]`, `city`, `linked_accounts[{platform,url,followers}]`, `rating`, `completed_deals`, `payout_balance`, `is_verified`
- **CompanyProfile** — `user_id`, `name`, `sphere`, `city`, `contact`, `rating`, `completed_lots`, `escrow_balance`
- **Lot** — `id`, `company_id`, `title`, `description`, `category`, `platforms[]`, `budget`, `deadline`, `requirements[]` (чек-лист), `status`, `chosen_response_id`, `created_at`
  - `status`: `draft → awaiting_payment → active → in_progress → completed / cancelled / disputed`
- **Response** — `id`, `lot_id`, `blogger_id`, `text`, `voice_file_id`, `status` (`pending`/`chosen`/`rejected`), `created_at`
- **Transaction (эскроу)** — `id`, `lot_id`, `company_id`, `blogger_id`, `amount`, `commission`, `status` (`held`/`released`/`refunded`), `psp_payment_id`, `created_at`, `released_at`
- **Review** — `id`, `lot_id`, `author_id`, `target_id`, `author_role`, `rating`, `text`, `created_at`
- **SavedSearch** («Хочу такой проект») — `id`, `blogger_id`, `categories[]`, `platforms[]`, `min_budget`, `city`, `notify_enabled`
- **DisputeCase** — `id`, `lot_id`, `opened_by`, `reason`, `resolution`, `decided_by`, `created_at`

## 5. Критический узел: платежи и эскроу

Это самая рискованная часть. Делать её надо рано и с открытыми глазами.

**Как реально устроен Telegram:** оплата идёт через Telegram Payments API по
`provider_token` от платёжного провайдера, и деньги поступают **продавцу**.
Telegram **не** держит деньги в эскроу и **не** делает автоматический сплит/выплату
блогеру. В Казахстане доступны эквайеры/шлюзы: **CloudPayments KZ**, **Freedom Pay**,
**Payselection** и др. (работают с юрлицами и ИП).

Из этого следует:
- **Приём денег от компании** — реализуемо: Telegram Payments + KZ-эквайер, либо
  чек-аут-ссылка PSP внутри Mini App. Нужен **юр. субъект** (ИП/ТОО).
- **Удержание + выплата блогеру за вычетом 10%** — нативно невозможно. Нужны либо
  ручные выплаты (MVP), либо marketplace/split-выплаты у PSP (позже).

**Юридическая структура (решает оркестратор + юрист).** Стандартная модель
маркетплейса — **агентский / комиссионный договор**: твои 10% — это агентская
комиссия, а остальное — транзит блогеру. Это обычно снимает необходимость в
лицензии платёжной организации, **но это нужно подтвердить для РК** (плюс AML и
налоги по выплатам физлицам). **Это спайк, который делается ДО написания кода
платежей.**

**Рекомендованный путь MVP (дёшево, быстро, низкий юр-риск для старта):**
- **v1 «ручной эскроу».** Компания оплачивает лот через PSP на счёт твоего ИП/ТОО →
  лот становится активным. После двустороннего подтверждения ты **выплачиваешь
  блогеру вручную** (банк/Kaspi) минус 10%. Приложение лишь ведёт состояния
  (`held`/`released`) и показывает балансы. Никакого кода авто-выплат — это
  позволяет **запуститься и проверить петлю**, пока решается автоматизация и
  юр-структура.
- **v2 «полу-авто».** Автоматизация выплат через payout/split API провайдера, когда
  объём оправдает и юр-модель подтверждена.

> **Решения оркестратора (до Фазы 4):** ИП/ТОО · выбор PSP · агентский договор +
> ревью юриста · политика отмены/возврата · формула разрешения споров.

## 6. Дорожная карта по фазам

Каждая фаза — вертикальный срез. Формат: **Цель · Claude Code делает · Оркестратор
решает/даёт · Готово, когда.**

### Фаза 0 — Каркас и аутентификация ⚠️ первым
- **Цель:** Mini App открывается из бота, проверяет `initData`, создаёт `User`; бот отвечает; есть репозиторий и деплой на staging.
- **Claude Code:** скелет фронта (шаблон), бот, бэкенд, БД, эндпоинт `/auth` с проверкой подписи `initData`, деплой.
- **Оркестратор:** создать бота в BotFather, привязать URL Mini App, дать хостинг/ключи, открыть приложение в Telegram.
- **Готово:** открываешь app из бота → видишь своё имя из Telegram → сервер подтверждает валидную сессию.

### Фаза 1 — Профили и роль
- **Цель:** регистрация: выбор роли, заполнение профиля (блогер: категории, город, ссылки на аккаунты; компания: название, сфера, город).
- **Готово:** оба профиля создаются/редактируются и сохраняются.

### Фаза 2 — Лоты: создание и лента
- **Цель:** компания создаёт лот (пока статус оплаты — заглушка-переключатель до Фазы 4); блогер видит ленту открытых лотов с фильтрами по категориям/площадкам.
- **Готово:** лот создаётся и появляется в ленте блогера с работающими фильтрами.

### Фаза 3 — Отклики и выбор
- **Цель:** блогер откликается (текст + загрузка голосового), напрямую владельцу писать нельзя; компания смотрит отклики по лоту и выбирает одного; после выбора показывается контакт в Telegram.
- **Готово:** полный цикл откликнуться → посмотреть → выбрать; выбранный блогер получает уведомление.

### Фаза 4 — Платежи (эскроу v1) ⚠️ зависит от спайка
- **Цель:** интеграция оплаты лота через PSP; лот становится активным только после оплаты; ведётся состояние эскроу; ручной инструмент выплат в админке.
- **Оркестратор (ДО фазы):** юр. субъект + договор PSP + ревью агентской модели юристом.
- **Готово:** компания проводит платёж (sandbox/реальный) → деньги отмечены как `held` → админ может пометить `released`.

### Фаза 5 — Закрытие сделки и отзывы
- **Цель:** двустороннее подтверждение → выплата (в v1 ручная) → обе стороны ставят оценку + отзыв; в профилях отображается рейтинг.
- **Готово:** цепочка подтверждение → выплата → отзыв работает; рейтинги обновляются.

### Фаза 6 — «Хочу такой проект» + уведомления
- **Цель:** сохранённый запрос + движок совпадений → push через бота при появлении подходящего лота.
- **Готово:** блогер сохраняет критерии; при создании совпадающего лота получает уведомление в Telegram.

### Фаза 7 — Админка, споры, модерация
- **Цель:** админ-панель: лоты/транзакции, разрешение споров, модерация профилей, ручные выплаты, базовая верификация (значок вручную).
- **Готово:** можешь вести операции и закрыть спор от начала до конца.

### Фаза 8 — Шлифовка и запуск
- **Цель:** пустые состояния, обработка ошибок, rate-limit, базовый анти-фрод, события аналитики, юр-страницы (оферта, политика), онбординг. Запуск в одном городе/вертикали.
- **Готово:** приложение можно показывать первым реальным пользователям.
